# Q02: Containerise a Flask Application Using Docker and Deploy the Image to Docker Hub

## Aim

To create a simple Flask web application, containerise it using Docker, test it locally, and push the Docker image to Docker Hub.

---

## Description

Flask is a lightweight Python web framework used to create web applications. Docker is a containerisation platform that packages the application and all its dependencies into a container image. Docker Hub is an online repository where Docker images can be stored and shared.

In this experiment:

* A Flask application is created.
* A Docker image is built using a Dockerfile.
* The container is run locally.
* The image is uploaded to Docker Hub.

---

# Requirements

* Python installed
* Docker Desktop installed and running
* Docker Hub account
* Internet connection

---

# Files Required

Create a folder named `flask-app` and inside it create the following files:

1. `app.py`
2. `requirements.txt`
3. `Dockerfile`

---

# Step 1: Create the Flask Application

Create a file named `app.py`

```python
from flask import Flask
app = Flask(__name__)

@app.route('/')
def home():
    return "Hello from Dockerised Flask!"

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
```

### Explanation

* `Flask(__name__)` creates the Flask application.
* `@app.route('/')` defines the home page.
* `host='0.0.0.0'` allows Docker to expose the application outside the container.
* `port=5000` runs the application on port 5000.

---

# Step 2: Create the Requirements File

Create a file named `requirements.txt`

```text
flask
```

### Explanation

This file contains the Python packages required by the application. Here, only Flask is needed.

---

# Step 3: Create the Dockerfile

Create a file named `Dockerfile`

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
EXPOSE 5000
CMD ["python", "app.py"]
```

### Explanation of Each Instruction

| Instruction                           | Purpose                                                   |
| ------------------------------------- | --------------------------------------------------------- |
| `FROM python:3.11-slim`               | Uses a lightweight Python 3.11 image as the base image    |
| `WORKDIR /app`                        | Sets `/app` as the working directory inside the container |
| `COPY requirements.txt .`             | Copies the requirements file into the container           |
| `RUN pip install -r requirements.txt` | Installs Flask inside the container                       |
| `COPY . .`                            | Copies all application files into the container           |
| `EXPOSE 5000`                         | Opens port 5000 for the Flask app                         |
| `CMD ["python", "app.py"]`            | Starts the Flask application when the container runs      |

---

# Step 4: Open Terminal / Command Prompt

Move into the project folder:

```bash
cd flask-app
```

---

# Step 5: Build the Docker Image

```bash
docker build -t <dockerhub-username>/flask-app:latest .
```

Example:

```bash
docker build -t abelmathew/flask-app:latest .
```

### Explanation

* `docker build` creates a Docker image.
* `-t` assigns a tag to the image.
* `latest` is the version tag.
* `.` means Dockerfile is present in the current directory.

Expected output:

```text
Successfully built <image-id>
Successfully tagged <dockerhub-username>/flask-app:latest
```

---

# Step 6: Check Whether the Image Was Created


You should see:


---

# Step 7: Run the Docker Container

```bash
docker run -d -p 5000:5000 <dockerhub-username>/flask-app:latest
```

Example:

```bash
docker run -d -p 5000:5000 abelmathew/flask-app:latest
```

### Explanation

* `-d` runs the container in background mode.
* `-p 5000:5000` maps:

  * local system port 5000
  * to container port 5000


You should see the running container in the list.

# Q03: Develop and Demonstrate a Simple Monolithic Application Model Using Flask

## Aim

To develop and demonstrate a simple monolithic application using Python Flask that performs CRUD operations on student data through REST API endpoints.

---

## Description

A monolithic application is a software architecture in which all parts of the application such as:

* User interface
* Business logic
* Data handling
* API routes

are combined into a single program and deployed as one unit.

In this experiment, a Student CRUD REST API is created using Flask. The application stores student records in memory using a Python dictionary. The following operations are supported:

1. Add a new student
2. View all students
3. Delete a student

Since all functionality is written in a single file (`app.py`), this is called a monolithic application.

---

# Software Required

* Python 3.x
* Flask
* Command Prompt / Terminal
* Text Editor or VS Code

---

# Theory

Flask is a lightweight Python web framework used to build web applications and APIs.

The given application contains:

* A Flask application object
* Routes for different operations
* In-memory student storage
* JSON request and response handling

The student data is stored in:

```python
students = {}
```

Each student is given a unique ID using:

```python
next_id = 1
```

---

# Program

Create a file named `app.py` and type the following code:

```python
from flask import Flask, request, jsonify

app = Flask(__name__)
students = {}
next_id = 1

@app.route('/students', methods=['GET'])
def get_students():
    return jsonify(list(students.values()))

@app.route('/students', methods=['POST'])
def add_student():
    global next_id
    data = request.get_json()
    student = {
        'id': next_id,
        'name': data['name'],
        'roll': data['roll']
    }
    students[next_id] = student
    next_id += 1
    return jsonify(student), 201

@app.route('/students/<int:sid>', methods=['DELETE'])
def delete_student(sid):
    students.pop(sid, None)
    return jsonify({'msg': 'Deleted'})

if __name__ == '__main__':
    app.run(debug=True)
```

# Q04 – Create a CI Pipeline Using GitHub Actions to Build and Push a Docker Image of a Python App to Docker Hub

## Aim

To create a Continuous Integration (CI) pipeline using GitHub Actions that automatically builds a Docker image of a Python application and pushes the image to Docker Hub whenever code is pushed to the `main` branch.

---

## Description

Continuous Integration (CI) is a DevOps practice in which code changes are automatically tested and built whenever developers push code to a repository.

In this experiment:

* A Python application is stored in a GitHub repository.
* GitHub Actions is used to automate the CI process.
* Whenever code is pushed to the `main` branch:

  1. GitHub checks out the source code.
  2. Logs in to Docker Hub.
  3. Builds a Docker image for the Python application.
  4. Pushes the image to Docker Hub.

This ensures that the latest version of the application is always available as a Docker image.

---

# Requirements

* GitHub account
* Docker Hub account
* Python application files
* Docker installed locally
* Git installed
* A GitHub repository containing the Python application

Example project structure:

```text
python-app/
│── app.py
│── requirements.txt
│── Dockerfile
└── .github/
    └── workflows/
        └── docker-publish.yml
```

---

# Sample Python Application

## app.py

```python
from flask import Flask

app = Flask(__name__)

@app.route('/')
def home():
    return "Hello from Python Docker App!"

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
```

## requirements.txt



---

# Procedure

2. Click **New Repository**.

4. Keep it public or private.
5. Save the repository.

## Step 3: Generate Docker Hub Access Token

1. Open Docker Hub.
2. Go to **Account Settings → Security**.
3. Click **New Access Token**.
4. Enter a token name such as `github-actions-token`.
5. Copy and save the generated token.

Note: Do not use your Docker Hub password directly. Use the access token.

---

## Step 4: Add Secrets to GitHub Repository

1. Open your GitHub repository.
2. Go to:

```text
Settings → Secrets and variables → Actions
```

3. Click **New repository secret**.
4. Add the following secrets:

| Secret Name     | Value                        |
| --------------- | ---------------------------- |
| DOCKER_USERNAME | Your Docker Hub username     |
| DOCKER_PASSWORD | Your Docker Hub access token |

These secrets are securely used inside the workflow.

---

## Step 5: Create GitHub Actions Workflow File

1. Inside your project, create the folder:

```text
.github/workflows/
```

2. Inside this folder, create a file named:

```text
docker-publish.yml
```

3. Add the following code:

```yaml
name: Build and Push Docker Image

on:
  push:
    branches: [ main ]

jobs:
  build-push:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build Docker image
        run: docker build -t ${{ secrets.DOCKER_USERNAME }}/python-app:latest .

      - name: Push to Docker Hub
        run: docker push ${{ secrets.DOCKER_USERNAME }}/python-app:latest
```

---

```yaml

    branches: [ main ]
```

* The workflow runs whenever code is pushed to the `main` branch.

```yaml

```

* Downloads the repository files into the GitHub Actions runner.

```yaml

```

* Logs in to Docker Hub using the stored secrets.

```yaml

```

* Builds a Docker image from the Dockerfile.

```yaml

```

* Uploads the Docker image to Docker Hub.

# Experiment Q05: Configure a Jenkins Pipeline to Automate Building and Deployment of a Dockerized Application from GitHub

## Aim

To configure a Jenkins pipeline that automatically fetches source code from a GitHub repository, builds a Docker image for the application, pushes the image to Docker Hub, and deploys the container automatically.

---

## Objective

* Install and run Jenkins inside a Docker container.
* Connect Jenkins with GitHub.
* Build a Docker image automatically using Jenkins.
* Push the image to Docker Hub.
* Deploy the Dockerized application automatically.

---

## Description

Jenkins is a Continuous Integration and Continuous Deployment (CI/CD) tool used to automate software development tasks. In this experiment, Jenkins is run inside a Docker container. The Docker socket of the host system is mounted inside Jenkins so that Jenkins can directly execute Docker commands.

The pipeline performs the following operations automatically:

1. Clones the application source code from GitHub.
2. Builds a Docker image.
3. Logs in to Docker Hub.
4. Pushes the image to Docker Hub.
5. Runs the image as a container for deployment.

Thus, whenever the source code is updated in GitHub, Jenkins can rebuild and redeploy the application automatically.

---

# Requirements

* Docker installed on the host system
* Docker Hub account
* GitHub account with a repository containing a Dockerized application
* Internet connection

---

# Procedure

## Step 1: Create a Persistent Volume for Jenkins

Create a Docker volume to store Jenkins configuration and data permanently.

```bash
docker volume create jenkins-data
```

### Explanation

* `docker volume create` creates a named volume.
* `jenkins-data` stores Jenkins jobs, plugins, and settings.
* Even if the Jenkins container is deleted, the data remains safe.

---

## Step 2: Start Jenkins Container

Run Jenkins as a Docker container.

```bash
docker run -d \
  --name jenkins \
  -p 8080:8080 -p 50000:50000 \
  -v jenkins-data:/var/jenkins_home \
  -v /var/run/docker.sock:/var/run/docker.sock \
  jenkins/jenkins:lts
```

### Explanation of Options

* `-d` → Runs the container in detached mode.
* `--name jenkins` → Assigns the name "jenkins" to the container.
* `-p 8080:8080` → Maps Jenkins web interface to port 8080.
* `-p 50000:50000` → Used for Jenkins agents.
* `-v jenkins-data:/var/jenkins_home` → Mounts the persistent volume.
* `-v /var/run/docker.sock:/var/run/docker.sock` → Allows Jenkins to access Docker on the host machine.
* `jenkins/jenkins:lts` → Uses the latest long-term support Jenkins image.

---

## Step 3: Obtain Jenkins Initial Admin Password

Get the automatically generated admin password.

```bash
docker exec jenkins cat /var/jenkins_home/secrets/initialAdminPassword
```

Copy the displayed password.

---

```text
http://localhost:8080
```

Paste the admin password and continue.

Then:

1. Select "Install Suggested Plugins".
2. Create a Jenkins admin user.
3. Complete the initial setup.

3. Open the "Available Plugins" tab.
4. Search for "Docker Pipeline".
5. Install the plugin.

### Purpose

The Docker Pipeline plugin allows Jenkins pipelines to execute Docker commands such as `docker build`, `docker push`, and `docker run`.

## Step 6: Install Docker CLI Inside Jenkins Container

Since Jenkins is running inside a container, Docker commands must be installed inside that container.

```bash
docker exec -u root jenkins bash -c "apt-get update && apt-get install -y docker.io"
```

### Explanation

* `docker exec` → Executes a command inside the Jenkins container.
* `-u root` → Runs the command as root user.
* `apt-get update` → Updates package information.
* `apt-get install -y docker.io` → Installs Docker CLI.

## Step 7: Create Jenkinsfile in GitHub Repository

Inside the root folder of the GitHub repository, create a file named `Jenkinsfile`.

```groovy
pipeline {
    agent any

    environment {
        IMAGE_NAME      = "myapp"
        DOCKER_HUB_USER = "your-dockerhub-username"
    }

    stages {

        stage('Clone') {
            steps {
                git branch: 'main',
                    url: 'https://github.com/your-username/your-repository.git'
            }
        }

        stage('Build Image') {
            steps {
                sh 'docker build -t $DOCKER_HUB_USER/$IMAGE_NAME:latest .'
            }
        }

        stage('Push to Hub') {
            steps {
                withCredentials([usernamePassword(
                    credentialsId: 'dockerhub-creds',
                    usernameVariable: 'DOCKER_USER',
                    passwordVariable: 'DOCKER_PASS'
                )]) {
                    sh '''
                    echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin
                    docker push $DOCKER_HUB_USER/$IMAGE_NAME:latest
                    '''
                }
            }
        }

        stage('Deploy') {
            steps {
                sh 'docker run -d -p 5000:5000 $DOCKER_HUB_USER/$IMAGE_NAME:latest'
            }
        }
    }
}
```

---

# Explanation of Pipeline Stages

## 1. Clone Stage

```groovy
git branch: 'main',
    url: 'https://github.com/your-username/your-repository.git'
```

This stage downloads the latest code from the GitHub repository.

---

```groovy

```

This command builds a Docker image using the Dockerfile present in the repository.


---


```groovy

```

Pushes the built image to Docker Hub.

---

## 4. Deploy Stage

```groovy

```

Runs the pushed image as a container.

The application becomes available at:

```text

```

# Experiment Q06: Use kubectl Commands to Create, Scale, and Manage Kubernetes Resources

## Aim

To learn how to use `kubectl` commands to create, scale, expose, inspect, and manage Kubernetes resources such as Deployments, Pods, and Services.

---

## Description

`kubectl` is the command-line tool used to interact with a Kubernetes cluster. It allows users to:

* Create deployments and pods
* Scale applications up or down
* View running resources
* Expose applications as services
* Inspect logs and pod details
* Delete resources when no longer required

In this experiment, an Nginx application is deployed into Kubernetes, scaled to multiple replicas, exposed using a service, and finally managed using both commands and a YAML manifest file.

---

# Procedure

## Step 1: Create a Deployment

A Deployment is created to run an Nginx container.

```bash
kubectl create deployment myapp --image=nginx:latest
```

### Verify the Deployment

```bash
kubectl get deployments
kubectl get pods
```

### Expected Output

* A deployment named `myapp` should be created.
* One pod should be running.

Example:

```bash
NAME    READY   UP-TO-DATE   AVAILABLE   AGE
myapp   1/1     1            1           20s
```

```bash
NAME                     READY   STATUS    RESTARTS   AGE
myapp-6f7b9c6d7b-abc12   1/1     Running   0          20s
```

---

## Step 2: Scale the Deployment

The deployment can be scaled to increase or decrease the number of running pod replicas.

### Scale Up to 3 Replicas

```bash
kubectl scale deployment myapp --replicas=3
```

Check the pods:

```bash
kubectl get pods
```

### Expected Result

Three pods should now be running.

Example:

```bash
NAME                     READY   STATUS    RESTARTS   AGE
myapp-6f7b9c6d7b-abc12   1/1     Running   0          2m
myapp-6f7b9c6d7b-def34   1/1     Running   0          15s
myapp-6f7b9c6d7b-ghi56   1/1     Running   0          15s
```

### Scale Back Down to 1 Replica

```bash
kubectl scale deployment myapp --replicas=1
```

Again check the pods:

```bash
kubectl get pods
```

Only one pod should remain active.

---

## Step 3: Expose the Deployment as a Service

To access the deployment from outside the cluster, expose it using a NodePort service.

```bash
kubectl expose deployment myapp --port=80 --type=NodePort
```

View the created service:

```bash
kubectl get services
```

Example:

```bash
NAME         TYPE       CLUSTER-IP      EXTERNAL-IP   PORT(S)        AGE
myapp        NodePort   10.96.145.211   <none>        80:30080/TCP   10s
```

The service exposes port 80 inside the cluster and maps it to a NodePort (such as 30080).

If using Minikube, access the service with:

```bash
minikube service myapp
```

---

## Step 4: Inspect Pod Details

To see detailed information about a pod:

```bash
kubectl describe pod <pod-name>
```

Example:

```bash
kubectl describe pod myapp-6f7b9c6d7b-abc12
```

This command displays:

* Pod IP address
* Container image
* Port details
* Events and status
* Resource information

---

## Step 5: View Pod Logs

To check the logs generated by the pod:

```bash

```

Example:

```bash
kubectl logs myapp-6f7b9c6d7b-abc12
```

This helps in debugging container issues.

---

## Step 6: Delete Resources

After completing the experiment, remove the deployment and service.

```bash
kubectl delete deployment myapp
kubectl delete service myapp
```

Verify deletion:

```bash
kubectl get all
```

The resources should no longer appear.

# Step 7: Create Deployment Using a YAML Manifest

Instead of creating resources through commands, Kubernetes resources can also be defined in a YAML file.

Create a file named `deployment.yaml`.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: nginx:latest
        ports:
        - containerPort: 80
```

Apply the YAML file:

```bash
kubectl apply -f deployment.yaml
```

Check all created resources:

```bash
kubectl get all
```

Example Output:

```bash
NAME                         READY   STATUS    RESTARTS   AGE
pod/myapp-7d5f5b7d5c-abc12   1/1     Running   0          20s
pod/myapp-7d5f5b7d5c-def34   1/1     Running   0          20s

NAME                    READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/myapp   2/2     2            2           20s
```

# Experiment 07: Create a GitHub Actions Pipeline to Automatically Build and Push a Docker Image of a Python App to Docker Hub

## Aim

To create a Continuous Integration (CI) pipeline using GitHub Actions that automatically builds a Docker image of a Python application and pushes the image to Docker Hub whenever code is pushed to the main branch.

---

## Description

GitHub Actions is a CI/CD service provided by GitHub that can automatically perform tasks such as testing, building, and deploying applications.

In this experiment:

* A Python application is stored in a GitHub repository.
* A Docker image is created from the application.
* GitHub Actions automatically builds the Docker image.
* The built image is pushed to Docker Hub.
* The process runs every time code is pushed to the `main` branch.

The pipeline uses:

* `actions/checkout` to download the repository code.
* `docker/setup-buildx-action` to enable Docker Buildx.
* `docker/login-action` to authenticate with Docker Hub.
* `docker/build-push-action` to build and push the Docker image.

---

# Requirements

Before starting, make sure the following are available:

1. GitHub account
2. Docker Hub account
3. Git installed on your system
4. A Python application with:

   * `app.py`
   * `requirements.txt`
   * `Dockerfile`
5. GitHub repository containing the project

---

# Sample Python Application

## app.py

```python
from flask import Flask

app = Flask(__name__)

@app.route('/')
def home():
    return "Hello from GitHub Actions CI/CD Pipeline!"

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
```

## requirements.txt

```text
flask
```

---

# Dockerfile

Create a file named `Dockerfile` in the project folder.

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . .

EXPOSE 5000

CMD ["python", "app.py"]
```

---

# Procedure

## Step 1: Create a GitHub Repository

1. Open GitHub.
2. Create a new repository.
3. Give the repository a name such as `python-docker-app`.
4. Upload the following files:

   * `app.py`
   * `requirements.txt`
   * `Dockerfile`

---

## Step 2: Create a Docker Hub Repository

1. Log in to Docker Hub.
2. Click "Create Repository".
3. Enter a repository name such as `python-app`.
4. Keep the repository public or private as required.
5. Save the repository.

---

## Step 3: Generate Docker Hub Access Token

1. Open Docker Hub.
2. Go to Account Settings → Security.
3. Click "New Access Token".
4. Give the token a name such as `github-actions-token`.
5. Copy and save the generated token.

This token will be used instead of your Docker Hub password.

---

## Step 4: Add Secrets to GitHub Repository

1. Open your GitHub repository.
2. Go to:
   `Settings → Secrets and variables → Actions`
3. Click `New repository secret`.
4. Add the following secrets:

| Secret Name     | Value                        |
| --------------- | ---------------------------- |
| DOCKER_USERNAME | Your Docker Hub username     |
| DOCKER_PASSWORD | Your Docker Hub access token |

---

## Step 5: Create GitHub Actions Workflow Folder

Inside your repository, create the following folder structure:

```text
.github/
└── workflows/
    └── build-push.yml
```

---

## Step 6: Create the Workflow File

Open `.github/workflows/build-push.yml` and add the following code:

```yaml
name: Docker Build & Push

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  docker:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v3

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and Push Docker Image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ secrets.DOCKER_USERNAME }}/python-app:latest
```

---

## Step 7: Commit and Push the Workflow

Run the following commands in the terminal:

```bash
git add .
git commit -m "Add GitHub Actions Docker build pipeline"
git push origin main
```

---

## Step 8: Verify the GitHub Actions Workflow

1. Open the GitHub repository.
2. Click the `Actions` tab.
3. Open the latest workflow run.
4. Verify that all steps are successful:

   * Checkout Repository
   * Set up Docker Buildx
   * Login to Docker Hub
   * Build and Push Docker Image

If successful, a green tick mark will appear.

---

## Step 9: Verify Image in Docker Hub

1. Open Docker Hub.
2. Open the `python-app` repository.
3. Check that the image tag `latest` is available.

Example image name:

```text
username/python-app:latest
```

---

# Explanation of Workflow

| Section                         | Purpose                                          |
| ------------------------------- | ------------------------------------------------ |
| `on:`                           | Specifies when the workflow should run           |
| `push`                          | Runs workflow when code is pushed to main branch |
| `pull_request`                  | Runs workflow when a pull request is created     |
| `actions/checkout@v3`           | Downloads the repository code                    |
| `docker/setup-buildx-action@v3` | Enables advanced Docker build support            |
| `docker/login-action@v3`        | Logs in to Docker Hub using secrets              |
| `docker/build-push-action@v5`   | Builds and pushes the Docker image               |
| `context: .`                    | Uses current project folder as build context     |
| `push: true`                    | Pushes the image to Docker Hub                   |
| `tags:`                         | Defines the Docker image name and tag            |

---

# Expected Output

After successful execution:

* GitHub Actions automatically builds the Docker image.
* The image is uploaded to Docker Hub.
* Every new push to the main branch updates the Docker image.

Example final image:

```text
mydockerhubusername/python-app:latest
```
